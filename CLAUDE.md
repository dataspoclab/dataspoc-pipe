# DataSpoc Pipe

## What is this project

DataSpoc Pipe is the **ingestion engine** of the DataSpoc platform. It reads data from sources (databases, APIs, spreadsheets) via Singer taps, converts to Parquet, and uploads to cloud buckets (S3, GCS, Azure). It is the first step in the data pipeline: raw data in → organized Parquet out.

## Core principle

**Simplicity.** Singer tap + Parquet + Bucket + Cron. Nothing more. No DAGs, no Airflow, no orchestration. One CLI command runs a pipeline. Cron schedules it.

## Architecture

```
[Data Source] → [Singer Tap (subprocess)] → stdout → [Pipe Engine] → transform(df) [if exists] → Parquet → [Cloud Bucket]
                                                          │
                                                    manifest.json (catalog)
                                                    state.json (incremental bookmarks)
                                                    logs/ (execution history)
```

- **Singer protocol**: taps emit SCHEMA, RECORD, STATE messages as JSON lines on stdout
- **Streaming conversion**: records buffered in batches (~10K), optionally transformed via user-defined `transform(df)`, converted to PyArrow Table, written as Parquet
- **Convention-based transforms**: if `~/.dataspoc-pipe/transforms/<pipeline_name>.py` exists and defines `def transform(df)`, it runs automatically per batch during ingestion. No config needed. Fail-safe: on error, raw data is written with a warning
- **Bucket as contract**: the directory structure in the bucket (`/raw/<source>/<table>/dt=YYYY-MM-DD/`) is the public interface that DataSpoc Lens and other tools depend on
- **Manifest**: JSON catalog at `/.dataspoc/manifest.json` tracks tables, schemas, timestamps, row counts

## Permissions & Access Control

**Design decision: DataSpoc does NOT implement RBAC or access control.** All permissions are delegated to the cloud provider's IAM (AWS IAM, GCP IAM, Azure AD).

- Pipe needs **WRITE** access to the destination bucket
- Each bucket should map to a permission level (e.g., `s3://company-public`, `s3://company-finance`, `s3://company-hr`)
- Users who lack IAM permission to a bucket simply cannot write to it — Pipe will fail with "Access Denied"
- Never implement authentication, authorization, or user management in Pipe code
- Credentials come from the environment (env vars, instance profiles, service accounts) — never from Pipe config

## Key design decisions

- **CLI-first**: all operations via `dataspoc-pipe` command. No GUI, no daemon, no server
- **Stateless**: no local database. All state (bookmarks, manifest, logs) lives in the bucket
- **Plugin-based**: taps are external Singer packages, executed as subprocesses
- **Built-in taps**: simple taps (google-sheets-public) are included, no extra install needed
- **Single-writer**: assumes one Pipe instance writes to a bucket at a time
- **Offline-friendly**: after config, works without internet (except for cloud upload)

## Directory structure

```
src/dataspoc_pipe/
  __init__.py           # Version
  cli.py                # Typer CLI — all commands
  config.py             # Pydantic models for pipeline config
  engine.py             # Core: Singer → Parquet → Bucket
  singer.py             # Singer protocol parser
  storage.py            # fsspec abstraction (S3, GCS, Azure, local)
  state.py              # Incremental state/bookmarks
  catalog.py            # Manifest + logs + status
  tap_templates.py      # Config templates for known taps
  builtin_taps/         # Built-in taps (no external install needed)
    google_sheets_public.py

tests/                  # pytest tests
docs/                   # Pre-dev docs + usage guides
examples/               # E2E demo, mock taps
```

## Conventions

- **Language**: Python 3.10+
- **CLI framework**: Typer
- **Config**: YAML (pipelines) + JSON (tap configs), validated with Pydantic
- **Package manager**: uv (preferred), pip compatible
- **Tests**: pytest, run with `pytest tests/ -v`
- **Cloud storage**: fsspec with s3fs/gcsfs/adlfs backends
- **Parquet**: PyArrow, compression zstd by default
- **CLI messages**: English (standardizing — some PT-BR may remain)

## How source configs work

```
~/.dataspoc-pipe/
  config.yaml           # Global defaults
  sources/              # One JSON per data source (generated by `add`)
    orders.json         # {"url": "https://docs.google.com/...", "sheet_name": "orders"}
    customers.json      # {"host": "db.example.com", "port": 5432, ...}
  pipelines/            # One YAML per pipeline (generated by `add`)
    orders.yaml         # source.tap + source.config path + destination bucket
    customers.yaml
  transforms/           # Optional Python scripts for per-batch data transforms
    orders.py           # def transform(df) → df — runs automatically if file exists
```

`dataspoc-pipe add <name>` creates both the source JSON (from template) and the pipeline YAML.

## Bucket convention (PUBLIC CONTRACT)

This is the interface between Pipe and Lens. Do not change without versioning.

```
<bucket>/
  .dataspoc/
    manifest.json                    # Catalog: tables, schemas, timestamps
    state/<pipeline>/state.json      # Incremental bookmarks
    logs/<pipeline>/<timestamp>.json # Execution logs
  raw/
    <source>/<table>/
      dt=YYYY-MM-DD/                 # Hive-style partitioning
        <table>_0000.parquet         # Data files
```

## What NOT to do

- Don't add DAG/orchestration features — cron is enough
- Don't add a web UI — this is CLI-only
- Don't implement data transformation — that's Lens's job
- Don't add streaming/real-time — this is batch-only
- Don't break the bucket convention — Lens depends on it
- Don't store secrets in pipeline YAML — use env vars or cloud IAM
